# PG-MCP 自然语言提问测试集

本文档用于测试 `pg-mcp` 的自然语言转 SQL 能力。  
数据源对应三个 fixture 数据库：

- `blog_small`
- `ecommerce_medium`
- `saas_crm_large`

建议测试方式：

1. 先做简单题，验证基础映射（单表、过滤、排序、聚合）。
2. 再做中等题，验证多表 JOIN、时间窗口、分组统计。
3. 最后做复杂题，验证 CTE、窗口函数、分层聚合、业务指标构造。

---

## blog_small

### 简单（L1）

1. `[B-L1-01]` 现在一共有多少用户？其中活跃用户有多少？
2. `[B-L1-02]` 按用户角色统计人数（reader、author、editor、admin）。
3. `[B-L1-03]` 最近 30 天创建的文章有哪些？按创建时间倒序给出标题和作者 ID。
4. `[B-L1-04]` 统计每种文章状态（draft/review/published/archived）的文章数量。
5. `[B-L1-05]` 浏览量最高的 20 篇文章标题、浏览量、发布时间分别是什么？
6. `[B-L1-06]` 每个分类下有多少篇文章？按文章数从高到低排序。
7. `[B-L1-07]` 每种语言（language_code）的文章数量和平均阅读时长是多少？

### 中等（L2）

1. `[B-L2-01]` 按作者统计：总发文数、已发布文章数、总浏览量，取前 20 名。
2. `[B-L2-02]` 哪些文章的评论数最多？返回文章标题、作者用户名、评论数（排除已删除评论）。
3. `[B-L2-03]` 最近 90 天内，每天新增评论数是多少？按日期升序输出。
4. `[B-L2-04]` 每个标签被多少篇文章使用？取使用次数前 30 的标签。
5. `[B-L2-05]` 按分类统计互动情况：文章数、评论数、reaction 数、平均浏览量。
6. `[B-L2-06]` 找出“高浏览低互动”文章：浏览量前 10%，但评论数和 reaction 数都偏低。
7. `[B-L2-07]` 每个作者最近 60 天的发文数、平均阅读时长、平均浏览量分别是多少？

### 复杂（L3）

1. `[B-L3-01]` 在每个分类中找出评论数最高的前 3 篇文章（需要分组内 TopN）。
2. `[B-L3-02]` 计算每位作者的“发布率”= published 文章数 / 总文章数，并按发布率排序。
3. `[B-L3-03]` 按月统计内容产出和增长：每月发文数、总浏览量，以及相对上月的变化率。
4. `[B-L3-04]` 找出疑似内容维护不足的文章：状态为 draft/review 且超过 30 天未更新。
5. `[B-L3-05]` 构造文章“互动率”指标 = (评论数 + reaction 数) / 浏览量，筛选浏览量>1000 的文章并排序。
6. `[B-L3-06]` 找出“稳定高产作者”：连续 3 个月每月至少发布 5 篇文章的作者。
7. `[B-L3-07]` 计算每个作者最近 90 天的综合活跃分（发文、被评论、被 reaction），并输出排名前 20。

## ecommerce_medium

### 简单（L1）

1. `[E-L1-01]` 当前有多少客户、多少商品、多少 SKU（product_variant）？
2. `[E-L1-02]` 按客户等级（new/silver/gold/platinum）统计客户数量。
3. `[E-L1-03]` 订单状态分布是什么？每种状态有多少单？
4. `[E-L1-04]` 最近 30 天每天的订单数和销售额分别是多少？
5. `[E-L1-05]` 库存最紧张的 50 个 SKU 是哪些（可用库存最少）？
6. `[E-L1-06]` 当前 open/pending 的客服工单有多少？按优先级分组统计。
7. `[E-L1-07]` 各流量渠道（website_session.channel）的会话数和转化会话数是多少？

### 中等（L2）

1. `[E-L2-01]` 每个客户等级的订单量、总收入、平均客单价分别是多少？
2. `[E-L2-02]` 统计近 90 天销量最高的商品（按件数和按销售额各给一份 Top20）。
3. `[E-L2-03]` 每个仓库的库存规模、低于补货点的 SKU 数量分别是多少？
4. `[E-L2-04]` 各承运商的履约表现：发货单量、平均配送时长、按时送达率。
5. `[E-L2-05]` 分析退货：按退货原因统计数量、平均退款金额、完成率。
6. `[E-L2-06]` 分析促销码效果：每个促销码的使用人数、使用次数、带来的折扣总额。
7. `[E-L2-07]` 对比不同设备（desktop/mobile/tablet）的会话数、转化率和平均点击深度。

### 复杂（L3）

1. `[E-L3-01]` 做 RFM 分群：按最近购买时间、购买频次、消费金额给客户打分并输出分群结果。
2. `[E-L3-02]` 计算“库存可覆盖天数”：结合近 14 天销量估算每个 SKU 的库存可销售天数。
3. `[E-L3-03]` 找出“高价值高风险客户”：LTV 高但近 60 天无下单且有未关闭工单。
4. `[E-L3-04]` 评估渠道质量：按渠道统计会话->下单转化漏斗、平均订单金额、退款率。
5. `[E-L3-05]` 识别异常会话：点击事件极多但无转化、且停留时长异常的 session。
6. `[E-L3-06]` 按商品类目计算“销售-退货净贡献”，筛选净贡献最低的类目和品牌组合。
7. `[E-L3-07]` 生成月度经营看板：订单数、收入、退款额、退货率、工单量、渠道转化率的月度趋势。

## saas_crm_large

### 简单（L1）

1. `[S-L1-01]` 按租户 tier 统计租户数量，并分 region_code 展示。
2. `[S-L1-02]` 按用户角色和状态统计用户数量（core.app_user）。
3. `[S-L1-03]` 销售线索（sales.lead）按状态分布如何？每种状态占比是多少？
4. `[S-L1-04]` 商机（sales.opportunity）按阶段分布：数量、总金额、平均赢单概率。
5. `[S-L1-05]` 支持工单（sales.ticket）按优先级和状态统计数量。
6. `[S-L1-06]` 订阅（billing.subscription）按状态统计数量、平均 seats、总 MRR。
7. `[S-L1-07]` 逾期发票（billing.invoice）有多少张、总金额多少？

### 中等（L2）

1. `[S-L2-01]` 按行业和账户 tier 分析 ARR：账户数、总 ARR、平均 ARR。
2. `[S-L2-02]` 复用 `sales.v_pipeline_by_stage` 输出当前 pipeline 结构，并按金额排序。
3. `[S-L2-03]` 复用 `sales.v_rep_activity_30d`，找出近 30 天最活跃的销售代表 Top50。
4. `[S-L2-04]` 复用 `billing.v_mrr_by_tier`，对比不同 tier 的活跃订阅与 MRR。
5. `[S-L2-05]` 复用 `billing.v_ar_aging`，输出应收账款账龄分布及金额占比。
6. `[S-L2-06]` 分析营销活动效果：按渠道统计成员数、发送数、打开率、点击率、转化率。
7. `[S-L2-07]` 统计平台集成稳定性：每个 integration 的任务总数、失败数、失败率。

### 复杂（L3）

1. `[S-L3-01]` 构建租户健康评分：综合逾期发票、失败同步任务、工单积压和 churn 信号。
2. `[S-L3-02]` 计算 lead->opportunity 转化率：按线索来源和渠道分组，比较不同 owner 的效果。
3. `[S-L3-03]` 做销售预测：按 region 和 expected_close_date（月）汇总加权 pipeline 金额。
4. `[S-L3-04]` 找出高 ARR 但健康分低的账户，输出风险原因（工单、逾期、使用下降等）。
5. `[S-L3-05]` 分析 seats 利用率：对比 subscription.seats 与 daily_account_metric.seats_in_use 趋势。
6. `[S-L3-06]` 评估 webhook 稳定性：按 endpoint 计算 5xx 比例、重试次数、平均响应时长。
7. `[S-L3-07]` 生成“营收与风险联合看板”：按月输出开票金额、回款情况、churn 风险账户数。

## 加分测试（跨难度通用）

1. 给同一个问题增加“最近 7 天 / 30 天 / 90 天”三个时间窗口，观察 SQL 是否稳定。
2. 给同一个问题增加“Top 10 / Top 100 / 不限制”三种结果规模，检查是否自动加 `LIMIT`。
3. 把问题改写成口语化、模糊化表达，验证模型是否能正确映射到表和字段。
4. 在问题中加入“排除取消订单、排除删除评论、只看活跃用户”等业务约束，验证过滤条件完整性。
5. 对复杂问题要求“先给 SQL 再给解释”，验证生成内容可读性和可执行性。
