# pg-mcp 整改清单与最小改造 PR 方案

> 日期：2026-02-25  
> 适用代码路径：`w5/pg-mcp`  
> 目标：以最小改动补齐关键能力缺口，并保持现有功能稳定

---

## 0. 背景与目标

当前 `pg-mcp` 已具备多数据库、SQL 只读校验、LLM 重试等基础能力，但在以下方面仍有明显缺口：

1. 可观测性不足（日志结构化不完整、`request_id` 不能全链路关联）。
2. 数据库连接韧性不足（连接失败缺乏明确重试/退避与降级策略）。
3. 缺少服务级限流与对应测试矩阵，难以验证高压场景行为。

本方案按优先级拆分为三个连续子 PR，确保每个阶段都可独立验收。

---

## 1. P0：可观测性最小闭环（结构化日志 + request_id 全链路）

### 1.1 目标

- 所有核心链路日志统一 JSON 结构。
- 每次 `query_database` 调用生成并携带 `request_id`，贯穿：
  - 入口
  - schema 获取
  - SQL 生成
  - SQL 校验
  - SQL 执行
  - 返回响应（成功/失败）

### 1.2 最小改造范围

- 新增/完善日志模块：`pg_mcp/utils/logging.py`
- 在启动阶段统一初始化日志：`pg_mcp/__main__.py`
- 在执行链路打点：`pg_mcp/server.py`, `pg_mcp/services/*.py`, `pg_mcp/security/validator.py`（最小必要打点）
- 在响应里补全 `request_id`（成功与失败）

### 1.3 验收标准

- 日志可被 JSON 解析。
- 任一请求可通过同一 `request_id` 串联至少 4 个阶段日志。
- 错误响应也包含 `request_id`。

---

## 2. P1：数据库连接重试/退避与健康降级

### 2.1 目标

- 数据库连接池创建失败时执行有界重试（指数退避 + 抖动）。
- 启动时允许“部分数据库不可用”并降级运行，不阻断全部服务。
- 请求访问不可用数据库时返回明确错误（而非模糊失败）。

### 2.2 最小改造范围

- `pg_mcp/services/executor.py`
  - 创建连接池重试逻辑
  - 记录不可用数据库状态
  - 暴露健康状态读取方法
- `pg_mcp/__main__.py`
  - 启动后记录健康摘要
  - 预加载 schema 时只处理可用数据库
- `pg_mcp/server.py`
  - 对不可用数据库返回明确错误码与详情

### 2.3 验收标准

- 单库失败不影响其他库可用。
- 重试次数与退避逻辑有单元测试覆盖。
- 错误响应可区分“库未配置”和“库不可连”。

---

## 3. P2：限流能力与测试矩阵

### 3.1 目标

- 补充服务级限流（最小实现）：
  - 并发请求上限（`max_concurrent_requests`）
  - 时间窗口请求上限（`rate_limit_per_minute`）
- 超限返回 `RATE_LIMIT_EXCEEDED`。

### 3.2 最小改造范围

- 配置扩展：`pg_mcp/config/settings.py`
- 限流器实现：`pg_mcp/services/rate_limiter.py`（新增）
- 接入调用入口：`pg_mcp/server.py`
- 异常定义补齐：`pg_mcp/exceptions/errors.py`
- 测试补齐：`tests/test_rate_limiter.py`（新增）与 `tests/test_tool.py` 增补超限场景

### 3.3 验收标准

- 并发超限与速率超限均可稳定复现。
- 返回错误码为 `RATE_LIMIT_EXCEEDED`，含可读错误信息。
- 新增测试全部通过。

---

## 4. 统一测试计划

### 4.1 必跑

- `uv run ruff check .`
- `uv run python -m pytest -q`

### 4.2 增补测试重点

- 日志结构与 `request_id` 透传
- 数据库连接重试行为（成功/耗尽）
- 部分数据库不可用时的降级行为
- 并发/速率限流行为

---

## 5. 风险与回滚

### 5.1 风险

- 新增日志字段可能影响既有日志消费端。
- 限流默认参数若设置过低可能影响吞吐。
- 连接降级策略可能改变默认库不可用时的行为预期。

### 5.2 回滚策略

- 限流参数支持默认关闭（0 表示不启用）。
- 连接重试参数使用保守默认值，必要时可调为 0 次重试。
- 任何阶段可通过单个提交回滚，不影响数据库 schema 数据。

---

## 6. 交付顺序

1. P0：可观测性闭环
2. P1：连接重试 + 健康降级
3. P2：限流 + 测试矩阵
4. 全量测试与结果记录
