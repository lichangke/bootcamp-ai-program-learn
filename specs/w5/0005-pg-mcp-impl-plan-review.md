## 审查总结

实现计划结构清晰、可执行性强，但**未完全覆盖 PRD 需求**，在需求追溯、可靠性和安全性方面存在若干高影响缺口。依赖关系图中未发现循环依赖，主要问题在于部分任务的依赖约束过度或不足，以及遗漏需求的依赖声明缺失。

## Critical 级别问题

### C1: PRD 需求追溯不完整 — FR-003 / FR-008 缺失

- **位置:** `§7 PRD 需求追溯矩阵` + `§3` 任务列表（无 FR-003 / FR-008 对应任务）
- **类别:** Traceability（需求追溯）
- **描述:** 追溯矩阵仅覆盖 FR-001/002/004/005/006/007/009 和部分 NFR。PRD 中的 FR-003（Schema 手动刷新 Tool）和 FR-008（OpenAI 结果有效性验证）在任务和验证中完全缺失。多个 NFR 也未追溯，尤其是 NFR-005（审计日志）、NFR-002（并发处理）、NFR-009（可观测性）。
- **建议:** 二选一：(1) 为缺失的 PRD 需求添加明确的实现阶段/任务；(2) 明确标记为 v1 范围外，添加延期路线图章节并说明对验收的影响。

### C2: 生命周期启动缺乏单库容错

- **位置:** `§3 / T7.2 实现 lifespan 生命周期管理`
- **类别:** Architecture（架构）
- **描述:** 启动序列为所有数据库统一初始化和发现 Schema；如果某个数据库连接失败，整个启动流程可能全局失败。这与 PRD NFR-007（"单个数据库不可用不影响其他数据库查询"）冲突。
- **建议:** 将数据库初始化/Schema 发现改为逐库容错：捕获单库错误、标记不健康状态、继续服务健康数据库，并在响应/日志中暴露就绪状态。

### C3: 缺少结构化审计日志实现任务

- **位置:** `§3`（`T0.1 utils/logging.py` 仅为骨架占位）, `§6.2 安全验收`
- **类别:** Security（安全）
- **描述:** 没有具体的实现任务覆盖结构化审计日志（NFR-005）：请求意图记录、生成的 SQL、校验结果、执行摘要、敏感字段脱敏策略。
- **建议:** 添加日志实现阶段/任务集：结构化 JSON 日志器、关联/请求 ID、敏感字段掩码、审计事件 Schema、保留策略钩子，以及验证无密钥泄露的测试。

## High 级别问题

### H1: 配置加载方式不完整

- **位置:** `§3 / T1.1 Settings`
- **类别:** Completeness（完整性）
- **描述:** FR-001 要求支持 YAML/JSON/ENV 配置、连接字符串格式、环境变量引用。T1.1 聚焦于 BaseSettings `.env` 加载，但未包含配置文件路径加载（`PG_MCP_CONFIG`）、JSON/YAML 解析、或显式的连接字符串处理流程。
- **建议:** 添加多源配置加载器优先级任务（`ENV > 文件默认值`）、`config.yaml/json` 解析、`connection_string` 支持，以及相应的验证测试。

### H2: Schema 发现范围窄于 PRD 要求

- **位置:** `§3 / T4.2 SchemaService` 和 `T4.3` 测试
- **类别:** Traceability（需求追溯）
- **描述:** FR-002 要求发现表、视图、索引、自定义类型和（可选）函数。T4.2 验收标准目前仅覆盖表/列/主键/注释（加系统 schema 排除），范围窄于 PRD。
- **建议:** 扩展 Schema 模型和发现查询以覆盖视图/索引/类型/函数，并相应扩展测试用例。

### H3: 错误码枚举不完整

- **位置:** `§3 / T1.2 异常体系`, `§3 / T8.1`, `§7 追溯矩阵`
- **类别:** Technical（技术）
- **描述:** T1.2 中的错误码枚举遗漏了 PRD/设计文档要求的错误码，如 `DB_CONNECTION_ERROR`、`RATE_LIMIT_EXCEEDED`，以及验证相关的错误码（FR-008 路径）。这导致错误契约不一致。
- **建议:** 将 ErrorCode 与 PRD §7.1 对齐，在 Tool 层定义映射表，并为所有公开错误响应添加契约测试。

### H4: 缺少 Prompt 注入防护和 LLM 数据安全任务

- **位置:** `§3 / T6.1 LLMService`, `T1.1 SecurityConfig`, `§6.2 安全验收`
- **类别:** Security（安全）
- **描述:** 计划包含 AST SQL 安全校验，但遗漏了 PRD NFR-005a/005b 控制：Prompt 注入检测、Schema/结果在发送给外部 LLM 前的脱敏处理、可选的"不出域模式"。
- **建议:** 添加安全子阶段：输入风险启发式检测、LLM 载荷清洗器、敏感列掩码策略、出站数据开关，以及专门的攻击场景测试。

### H5: SQL 校验器测试矩阵覆盖不足

- **位置:** `§3 / T2.2 SQL 校验器测试矩阵`
- **类别:** Testing（测试）
- **描述:** PRD 要求禁止的操作包括 GRANT/REVOKE/EXECUTE（以及 DDL/DML 全覆盖）。当前测试矩阵未显式覆盖这些，且缺少多语句攻击尝试（`SELECT ...; DROP ...`）的覆盖。
- **建议:** 添加 GRANT/REVOKE/EXECUTE/LOCK/COPY/多语句载荷的显式测试用例，并断言确定性的拒绝原因。

### H6: 缺少数据库连接重试策略

- **位置:** `§3 / T3.1 SQLExecutor`, `T7.2 startup`
- **类别:** Risk（风险）
- **描述:** PRD NFR-006 要求数据库连接失败自动重试/降级行为。计划仅为 DeepSeek 设计了重试（T6.1），但没有数据库初始化重试/退避策略。
- **建议:** 为连接池创建和 Schema 发现添加有界退避重试策略；不健康数据库降级处理，不导致全局失败。

## Medium 级别问题

### M1: 任务依赖图过度串行化

- **位置:** `§4 任务依赖关系图`（`T4.1` 串联在 `T3.1` 之后）
- **类别:** Dependency（依赖）
- **描述:** 依赖图将一些逻辑上独立的任务串行化（例如 Schema 数据模型可以在执行器完成之前构建）。这可能拖慢交付并模糊关键路径。
- **建议:** 细化 DAG：允许 `T4.1` 和 `T5.*` 更早并行启动，仅保留真正的运行时依赖。

### M2: CTE 顶层语句检查可能不准确

- **位置:** `§3 / T2.1 _check_top_level_statement` 指导
- **类别:** Technical（技术）
- **描述:** 指导假设顶层可能是 `With`；但在 SQLGlot 中，许多 CTE 查询解析为带 `with` 参数的 `Select`。如果按字面实现，CTE 处理可能脆弱/不正确。
- **建议:** 针对锁定的 SQLGlot 版本验证 AST 行为，在编码规则逻辑之前为多种 CTE 模式定义验收测试。

### M3: LIMIT 回退策略存在安全隐患

- **位置:** `§3 / T3.1 LIMIT 回退（"AST 失败回退字符串拼接"）`
- **类别:** Security（安全）
- **描述:** AST 解析失败时的字符串追加回退（`... LIMIT n`）可能对边缘 SQL 形式和多语句边界情况产生不安全/无效的语义。
- **建议:** 优先在 AST 解析失败时采用 fail-closed 策略，或严格限制回退仅适用于已验证的单语句 SELECT 模式。

### M4: 缺少并发测试计划

- **位置:** `§3` 测试整体（`T3.3`, `T8.2`, `T9.2`）
- **类别:** Testing（测试）
- **描述:** 没有针对 PRD NFR-002 并发处理的显式测试计划（并行查询负载、连接池压力、LLM 调用限流）。
- **建议:** 添加异步并发测试（如 20-100 并行请求）、连接池耗尽行为测试、延迟/错误率断言。

### M5: 风险登记表遗漏关键运维/安全风险

- **位置:** `§5 风险与缓解`
- **类别:** Risk（风险）
- **描述:** 风险表遗漏了关键的运维/安全风险：Prompt 注入、出站数据泄露、部分数据库故障行为、可观测性盲区。
- **建议:** 扩展风险登记表，为这些场景添加触发条件/影响/负责人/缓解措施/监控。

### M6: Docker 部署缺少安全加固

- **位置:** `§3 / T9.3 Docker 部署`
- **类别:** Security（安全）
- **描述:** Docker 计划过于简单，缺少安全加固（非 root 用户、最小化复制、依赖锁定、镜像精简）。
- **建议:** 添加 Docker 加固任务：`.dockerignore`、非 root 运行时用户、锁定依赖版本、多阶段构建、CI 中的容器安全扫描。

## Low / Info 级别问题

### L1: 缺少显式的 v1 范围边界声明

- **位置:** `§1 实现概述`, `§7 追溯矩阵`
- **类别:** Architecture（架构）
- **描述:** 计划看起来针对 v1 子集，但未显式声明范围边界。这使得遗漏看起来像是意外而非有意为之。
- **建议:** 添加显式的"范围 / 非目标 / 延期到 v2"章节，列出有意排除的 FR/NFR。

### I1: 测试覆盖率目标不一致

- **位置:** `§6.3 质量验收` vs 模块级目标（`T1.3 90%`, `T2.2 95%`）
- **类别:** Testing（测试）
- **描述:** 覆盖率目标不一致（模块级严格、全局宽松），缺乏理由说明；这可能掩盖不均匀的质量。
- **建议:** 定义统一的覆盖率策略（全局 + 关键模块最低值）并在 CI 中强制执行。

---

## 依赖关系审查结论

当前 DAG 中**未发现循环依赖**。主要问题是：
1. 部分任务的依赖约束过度串行化（可并行的任务被不必要地串联）
2. 遗漏需求（FR-003、FR-008、NFR-005 等）导致缺少对应的依赖声明
